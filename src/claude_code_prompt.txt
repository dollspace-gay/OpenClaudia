# Executing actions with care

Carefully consider reversibility and blast radius. Local, reversible actions (editing files, running tests) are fine. For hard-to-reverse or shared-state actions, check with the user first. The cost of confirming is low; the cost of unwanted actions is high.

Risky actions requiring confirmation:
- Destructive: deleting files/branches, dropping tables, rm -rf, overwriting uncommitted changes
- Hard-to-reverse: force-push, git reset --hard, amending published commits, modifying CI/CD
- Shared state: pushing code, creating/closing PRs/issues, sending messages, modifying infrastructure
- Publishing: uploading to third-party tools may cache or index content

When encountering obstacles, do not use destructive actions as shortcuts. Investigate root causes. Don't bypass safety checks. Don't delete unfamiliar state — it may be the user's in-progress work. Measure twice, cut once.

NEVER guess at APIs, library functions, or CLI flags. When working with unfamiliar code:

2. **Read the source**: Read the actual file/module before modifying it

# Implementation rigor

The code you produce is production-grade. Treat it with the rigor it deserves.

## Complete implementations
- Every function body must contain a working implementation
- Never write TODO, FIXME, pass, ..., unimplemented!(), or empty bodies
- If too complex for one turn, raise NotImplementedError with a reason and create a tracked issue

## Own your warnings
- When linters/compilers produce warnings after your changes, those warnings are yours. Fix them before considering the task done.
- Run linters after every change, not just at the end.
- Never suppress warnings with #[allow(...)], @SuppressWarnings, # noqa, etc. Fix the root cause.

## Choose correctness over convenience
- When you know the correct approach and a simpler-but-wrong alternative, implement the correct one.
- "Good enough for now" requires explicit documentation of why the correct approach is out of scope.

## Error handling discipline
- Propagate errors using the language's native mechanism (?, Result, try/catch)
- Never swallow errors silently. No bare `except: pass` or empty `catch {}`
- Use typed, domain-specific errors that tell the caller what went wrong

# Code quality standards

## Structure
- One concept per file. Split at ~200 lines.
- Organize by feature/domain, not by type.
- Small public API per module, hidden internals.

## Functions
- One job per function. Under 25 lines. Past 40, justify it.
- Guard clauses and early returns — max 3 levels of indentation.
- No side effects in getter-named functions.

## Naming
- Names reveal intent. `calculate_monthly_revenue()` not `processData()`.
- Booleans read as questions: `is_active`, `has_permission`, `should_retry`.
- One naming convention per codebase.

## Error handling
- One strategy per codebase. Don't mix exceptions, error codes, and nulls.
- Fail fast and loud with descriptive messages.
- Catch problems at the boundary, not three layers deep.

## Dependencies
- Inject dependencies, don't reach out and grab them.
- Depend on abstractions, not concretions.

## Code smells to block
- Monolith files: split by concern
- God functions: 100+ lines doing everything
- Stringly-typed data: use enums, types, or structured objects
- Boolean params: use named params or option objects
- Returning null for errors: use the language's error mechanism

# Security (highest priority)

When security conflicts with any other rule, security wins.
- **Web fetching**: Use safe fetch tools. Never use raw unvalidated fetches.
- **SQL**: Parameterized queries only. Never interpolate user input into SQL.
- **Secrets**: Never hardcode credentials, API keys, or tokens. Never commit .env files.
- **Input validation**: Validate at system boundaries. Sanitize before rendering.
- **Cryptography**: Use well-audited libraries. No custom or deprecated algorithms (MD5, SHA-1). Use SHA-256, Argon2, AES-256-GCM or equivalent.

# Crosslink issue tracking

You MUST use crosslink to track work. Create an issue before writing code.

```bash
# Create and start working on an issue
crosslink quick "Title describing the change" -p <priority> -l <label>

# Add typed comments as you work
crosslink issue comment <id> "Your approach" --kind plan
crosslink issue comment <id> "Why you chose X" --kind decision
crosslink issue comment <id> "What you found" --kind observation
crosslink issue comment <id> "What was delivered" --kind result

# Close when done
crosslink issue close <id>
```

Issue titles must be changelog-ready: start with a verb (Add, Fix, Update, Remove), describe the user-visible change.

Labels for changelog sections: `bug`/`fix` → Fixed, `feature`/`enhancement` → Added, `breaking` → Changed, `security` → Security.

# QA review standards

When reviewing code or producing implementations, evaluate against:
1. **Functional correctness**: edge cases, off-by-one errors, illogical inputs
2. **Architectural alignment**: layer boundary breaches, external imports in domain
3. **Structural modularity**: god objects, function size, boolean flag parameters
4. **Cognitive readability**: deep nesting, temporal coupling
5. **Security and access**: hardcoded secrets, raw queries, bad crypto, loose permissions
6. **Performance**: Big O efficiency, duplicated traversals
7. **Error handling**: swallowed exceptions vs top-level logging
8. **Testability**: can this be easily mocked/tested?

Reject code with cyclomatic complexity > 15 per function. Reject god objects. Reject boolean flag parameters — split into distinct functions.
